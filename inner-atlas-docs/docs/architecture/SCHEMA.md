# Inner Atlas — Database Schema

## Storage Architecture

**Default:** SQLite via `expo-sqlite`, local on-device.  
**Optional:** Supabase PostgreSQL (cloud sync, opt-in only — see TECH_STACK.md).

The schema is identical in both modes. All tables are created locally on first app launch. If a user enables cloud sync, the same schema is instantiated in Supabase with RLS added.

## SQLite Setup Notes

- Database file: `inner-atlas.db`
- Initialized on first launch via `docs/architecture/db-init.sql` (generated by Claude Code in Phase 0)
- All queries use parameterized statements — no string interpolation
- `user_id` fields exist for schema compatibility with eventual cloud sync but are ignored in local-only mode (single-user device)
- Foreign key enforcement: `PRAGMA foreign_keys = ON`

## Core Tables

### profiles
```sql
id TEXT PRIMARY KEY,
display_name TEXT,
role TEXT DEFAULT 'individual',  -- individual | client | therapist
onboarding_complete INTEGER DEFAULT 0,
assessment_complete INTEGER DEFAULT 0,
settings_json TEXT DEFAULT '{}',
created_at TEXT,
updated_at TEXT
```

### parts
```sql
id TEXT PRIMARY KEY,
name TEXT NOT NULL,              -- original assessment working title — never changes
custom_name TEXT,                -- user-assigned via Naming Moment or Refine
display_name TEXT GENERATED ALWAYS AS (COALESCE(custom_name, name)) VIRTUAL,
type TEXT NOT NULL,              -- manager | firefighter | exile | self
backend_classification TEXT,     -- NEVER shown in UI
intensity INTEGER DEFAULT 5,     -- 1-10
activation_status TEXT DEFAULT 'moderate',  -- high | moderate | low
icon_id TEXT,
color_hex TEXT,
position_x REAL,
position_y REAL,
discovered_via TEXT,             -- first_mapping | mini_[name] | manual | trailhead
status TEXT DEFAULT 'named',     -- named | shadowed | unknown
is_elaborated INTEGER DEFAULT 0,
is_refined INTEGER DEFAULT 0,
created_at TEXT,
updated_at TEXT
```
_Note: `backend_classification` stored for clinical inference logic only. Never exposed in any UI component._

### part_profiles
```sql
part_id TEXT PRIMARY KEY REFERENCES parts(id),
triggers_json TEXT,
manifestations_json TEXT,        -- {cognitive, emotional, somatic, behavioral}
somatic_locations_json TEXT,
developmental_history TEXT,
earliest_memory TEXT,
part_perspective TEXT,
burden_description TEXT,
gift_description TEXT,
feel_towards TEXT,
appearance_json TEXT,            -- {color, shape, texture, movement, temperature, sound, age, image_uri}
custom_attributes_json TEXT,
updated_at TEXT
```

### part_relationships
```sql
id TEXT PRIMARY KEY,
part_a_id TEXT REFERENCES parts(id),
part_b_id TEXT REFERENCES parts(id),
relationship_type TEXT,          -- harmonious | conflicting | protective | neutral | polarized
direction TEXT,                  -- a_to_b | b_to_a | mutual
strength INTEGER DEFAULT 5,      -- 1-10
status TEXT DEFAULT 'confirmed', -- confirmed | inferred | shadowed
created_at TEXT,
updated_at TEXT
```

## Data Export

The export function (Settings → Export My Data) produces a single JSON file containing all tables as arrays. This file:
- Is the user's full data backup
- Can be imported on a new device to restore data
- Can be selectively shared — user opens the file and chooses what to include

### Clinical Metrics Export (separate)

Produces a structured summary containing ONLY:
- Assessment cluster scores (A/B/C/D) and dominant inferences
- Self-energy trend (overall_percentage over time)
- Activation frequency by part type
- Milestone keys earned
- Technique completion counts

**Explicitly excluded:** all part names, custom_name values, dialogue content, profile text fields, naming moment responses, trailhead descriptions, update content.

Client sends this file through their own channel. It never transmits automatically.

## Cloud Sync Schema (opt-in only)

When a user enables cloud sync, the identical schema is created in Supabase with:
- `user_id UUID REFERENCES auth.users` added to every table
- Row Level Security enabled on all tables
- All local SQLite data migrated via export/import on first sync
- Ongoing sync via timestamp-based change detection

This is a Phase 5+ feature. Do not build in Phase 0-3.

## expo-secure-store Keys

These are stored in the device's secure enclave (not in SQLite). Never in plaintext.

| Key | Contents |
|---|---|
| `db_encryption_key` | SQLCipher encryption key (hex string, generated on first launch) |
| `app_lock_pin_hash` | bcrypt hash of user's PIN (if app lock enabled) |
| `app_lock_settings` | JSON: `{enabled, method, timeout_minutes}` |
| `last_backup_at` | ISO timestamp of most recent successful backup |

These keys persist across app updates but are wiped on app uninstall.  
**Never read or log these values in any UI component or console statement.**
